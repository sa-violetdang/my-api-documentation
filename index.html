Pagination spec
GET/pagination

This document describes the common standard for handling 2 query parameters: limit and page used for endpoints that return lists

  • limit (number, optional)

    • Meaning: Maximum number of elements returned per page
    • Default: Environment variable value [DEFAULT_PAGE_SIZE] (if [Parameters.limit] is not specified, use default value)
    • Min: 1
    • Max: Environment variable value [MAX_PAGE_SIZE] (if exceeded, return 422 error)

  • page (number, optional)

    • Meaning: Specify the page number to return. (starts from 1)
    • Default: 1 (if [Parameters.page] is not specified, use default value)
    • Min: 1

  • Among records that meet search conditions, only return records with order from (([page]-1) * [limit])+1 to record number [page] * [limit]

    • Example: [Parameters.page] = 3 & [Parameters.limit] = 20 → Return records from 41st to 60th

userSearch
GET/users{?name,email,ageFrom,ageTo,page,limit}

  • Search for users in the system based on specified criteria

Search Processing ¶

  • [Parameters.name]: Add search condition [users.name] LIKE ‘%[Parameter.name]%’

  • [Parameters.email]: Add search condition [users.email] LIKE ‘%[Parameter.email]%’

  • [Parameters.ageFrom]: Add search condition [users.age] >= %[Parameter.ageFrom]%

  • [Parameters.ageTo]: Add search condition [users.age] <= %[Parameter.ageTo]%

  • [Parameters.page], [Parameters.limit]: See Common pagination spec

  • Create a new user based on the received information

Validation ¶

  • Check required:

    • [Request.name]
    • [Request.email]

  • Check length:

    • [Request.name]: max 50
    • [Request.email]: max 255
    • [Request.age]: max 3

  • Check format:

    • [Request.email]: RFC standard email format
    • [Request.age]: positive integer only

    • [name] = [Request.name]
    • [email] = [Request.email]
    • [age] = [Request.age]

orderCreate
POST/orders

  • Create a new order associated with the logged-in user based on received information

Validation ¶

  • Check required:

    • [Request.details] (array must not be empty)

  • Check for each detail item:

    • [Request.details.product_id]: required, must be valid product ID
    • [Request.details.unit_price]: required, positive number
    • [Request.details.quantity]: required, positive integer

Save Processing ¶

  • Insert new record into [orders] table

    • [total_amount] = Sum of [Request.details.unit_price]*[Request.details.quantity] for all [Request.details] objects
    • [created_by] = [users.id] of logged-in user
    • [status] = ‘pending’

  • Insert new records into [order_details] table equal to number of [Request.details] objects received

    • [order_id] = newly created [orders.id] from above
    • [product_id] = [Request.details.product_id]
    • [unit_price] = [Request.details.unit_price]
    • [quantity] = [Request.details.quantity]
    • [amount] = [Request.details.unit_price]*[Request.details.quantity]

orderUpdateStatus
PATCH/orders/{id}/status

  • Update the order status to the specified status

Validation ¶

  • Get [orders] record that satisfies all conditions below. If no record exists, return 404 error

    • [orders.id] === [Parameters.id]
    • [orders.deleted_at] === null

  • Check required:

    • [Request.status]

  • Check value list:

    • If [Request.status] NOT IN (‘pending’,‘confirmed’,‘processing’,‘packing’,‘shipped’,‘delivered’,‘completed’,‘canceled’), return 422 error

Save Processing ¶

  • Update [orders] record where [orders.id] === [Parameters.id]
    • [status] = [Request.status]
